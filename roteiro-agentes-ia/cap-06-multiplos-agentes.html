6. Dividindo responsabilidades entre múltiplos agentes especializados

Arquitetura: quando um único agente deixa de ser suficiente e como desenhar um sistema com vários.

🎯 Objetivos de aprendizagem

  • Entender o que são subagentes e como eles rodam em contexto isolado.
  • Reconhecer os sinais de que uma tarefa deveria ser dividida entre agentes especializados.
  • Conhecer os padrões "Dynamic Workflows" e "Performance Outcomes" (2026).
  • Entender o spawning hierárquico (agentes que criam outros agentes) e seus limites.
  • Diferenciar paralelismo de sequenciamento na orquestração de agentes.

Explicação conceitual

Até aqui você usou o Claude Code como um agente único conversando com você. Mas em projetos maiores, uma única janela de contexto vira um gargalo: pesquisar em 40 arquivos, rodar testes, ler logs longos e ainda implementar código na mesma conversa faz o contexto "encher" rápido, degradando a qualidade das respostas.

A solução é usar subagentes: agentes definidos como arquivos Markdown em .claude/agents/, cada um com seu próprio system prompt, seu próprio conjunto de tools permitidas e, opcionalmente, seu próprio modelo. Cada subagente roda em um contexto isolado — ele não carrega o histórico da conversa principal, só recebe a tarefa que lhe foi delegada e devolve um resultado resumido. Isso é ótimo para três situações:

  • Pesquisa paralela — várias buscas independentes (ex: "onde está a lógica de autenticação", "onde estão os testes de pagamento") rodando ao mesmo tempo.
  • Tarefas ruidosas — ler logs enormes, resultados de busca extensos ou saídas de comando verbosas; o subagente processa o ruído e devolve só a conclusão.
  • Modificações paralelas independentes — editar módulos que não se tocam (ex: frontend e backend de uma feature) ao mesmo tempo.

Em 2026, dois padrões se consolidaram no uso avançado de subagentes. O primeiro é Dynamic Workflows: um agente líder (orquestrador) analisa a tarefa, monta um plano e dispara — sozinho, sem intervenção humana a cada passo — dezenas ou até centenas de subagentes em paralelo, cada um cuidando de uma fatia do problema. O segundo é Performance Outcomes: um agente "grader" (avaliador), separado do agente que fez o trabalho, revisa o resultado contra um rubric (critério de qualidade) e devolve o subagente para revisão até o padrão ser atingido — uma espécie de loop de controle de qualidade automatizado.

Também é possível o spawning hierárquico: um agente pai cria agentes filhos, que por sua vez podem criar seus próprios filhos, até um limite prático de 5 níveis. Isso é útil para decompor tarefas grandes que tocam múltiplos módulos — por exemplo, um agente de "migração de API" que cria um filho por serviço afetado, e cada filho cria netos para lidar com cada endpoint. Passar desse limite tende a gerar overhead de coordenação maior que o ganho de paralelismo.

Exemplos práticos

Você não precisa "programar" a orquestração — basta pedir em linguagem natural e o Claude Code decide quando delegar, com base na description de cada subagente disponível em .claude/agents/. Exemplo de prompt para o agente principal: 

Preciso migrar todos os componentes de src/components/legacy para o novo
padrão de hooks. São 12 componentes independentes entre si.
Use subagentes em paralelo, um por componente, e depois consolide um
resumo do que foi alterado em cada um.

Estrutura típica de projeto com múltiplos agentes especializados (cada um será detalhado mecanicamente no capítulo 7): 

.claude/
  agents/
    arquiteto.md        # planeja mudanças estruturais
    implementador.md    # escreve código a partir de um plano aprovado
    revisor-codigo.md   # revisão de qualidade e padrões
    qa-tester.md        # escreve e roda testes
    seguranca.md        # audita vulnerabilidades e segredos expostos
    devops.md           # cuida de CI/CD, infra, deploy

Fluxo de trabalho recomendado

  1. Comece sempre com um agente único até sentir o contexto ficar pesado ou o escopo se tornar claramente paralelizável.
  2. Identifique papéis, não pessoas: "quem planeja", "quem implementa", "quem audita" — não confunda divisão de papéis com divisão arbitrária de arquivos.
  3. Dê ao agente orquestrador uma instrução explícita de quando paralelizar e quando esperar um subagente terminar antes de iniciar outro (dependências sequenciais).
  4. Use um agente revisor/grader separado do agente que implementou — nunca deixe quem escreveu o código ser o único a validar a própria qualidade.
  5. Consolide sempre um resumo final para o humano; não deixe os detalhes de cada subagente poluir a conversa principal.

Erros comuns

Erro comum: dividir agentes por arquivo em vez de por responsabilidade. Isso cria múltiplos agentes fazendo a mesma coisa (implementar) sem nenhum ganho de qualidade, só de velocidade — e ainda corre risco de conflito quando os arquivos na verdade se relacionam.
Erro comum: paralelizar tarefas que têm dependência real entre si (ex: implementar a API antes do frontend que a consome). O resultado é retrabalho, porque um subagente termina antes de saber a interface que o outro definiu.
Erro comum: ultrapassar os 5 níveis de spawning hierárquico "porque dá para fazer". O overhead de coordenação cresce mais rápido que o ganho — geralmente 1-2 níveis já resolvem a maioria dos casos reais.
Mão na massa: pegue uma tarefa do seu backlog que toque pelo menos 3 arquivos não relacionados entre si (ex: 3 componentes de UI independentes). Peça ao Claude Code para usar subagentes em paralelo, um por arquivo, e depois consolidar um resumo. Compare o tempo e a qualidade do resultado com fazer a mesma tarefa em uma única conversa sequencial.

Resumo do capítulo

Subagentes rodam em contexto isolado e resolvem o problema de contexto poluído e trabalho paralelizável. Os padrões Dynamic Workflows (orquestração automática em larga escala) e Performance Outcomes (grader separado validando contra um rubric) representam o estado da arte em 2026. O spawning hierárquico permite decompor tarefas grandes em até 5 níveis, mas o critério para dividir deve sempre ser responsabilidade e paralelizabilidade real — nunca divisão arbitrária de arquivos. O próximo capítulo mostra a mecânica de criar cada um desses agentes na prática.
