13. Demonstração completa — um projeto construído com múltiplos agentes

Estudo de caso ponta a ponta: o endpoint de cadastro de usuário do user-api, construído por cinco agentes especializados e um humano no comando.

🎯 Objetivos de aprendizagem

  • Ver, na prática, como Spec-Driven Development e multi-agentes se conectam em um fluxo real.
  • Entender a divisão de responsabilidades entre agente de planejamento, implementação, revisão, QA e segurança.
  • Analisar exemplos concretos de spec.md, plan.md e um diff revisado.
  • Identificar os pontos onde o humano precisa intervir e decidir.

Explicação conceitual

Este capítulo é diferente dos anteriores: em vez de explicar um conceito isoladamente, vamos narrar um projeto real do início ao fim, aplicando tudo o que foi visto no curso. O cenário: o user-api, nosso projeto Python/FastAPI recorrente, precisa do endpoint POST /usuarios para cadastro de usuário — simples o suficiente para caber em um capítulo, mas realista o suficiente para ter todas as arestas que aparecem em projetos de verdade: validação de entrada, persistência via repositório, testes e uma superfície de ataque real (e-mail duplicado, dados inválidos, mensagens de erro que vazam informação).

Vamos usar cinco papéis de agente, cada um com uma responsabilidade única — o mesmo princípio de especialização que vimos nos capítulos 5, 6 e 9:

Exemplos práticos

1. O pedido inicial e o spec.md

O humano começa com um pedido em linguagem natural, propositalmente não muito detalhado — é papel do arquiteto-back extrair as perguntas certas e formalizar: 

Você: Preciso do endpoint POST /usuarios no user-api, para cadastro
de usuário com nome e e-mail. Use o arquiteto-back para gerar
a spec antes de qualquer código.

O arquiteto-back devolve algo como:

# spec.md — Cadastro de usuário (POST /usuarios)

## Objetivo
Fornecer um endpoint REST para cadastrar um novo usuário no
user-api, validando nome e e-mail e persistindo via repositório.

## Escopo
- POST /usuarios recebendo { nome, email }
- Validação de nome (não vazio) e e-mail (formato válido)
- Persistência via repositório de usuários (PostgreSQL)
- Rejeição de e-mail duplicado

## Fora de escopo (v1)
- Autenticação/login (endpoint futuro)
- Confirmação de e-mail por link
- Edição/remoção de usuário

## Requisitos não funcionais
- Arquitetura hexagonal: domínio em src/domain/, adapter HTTP em
  src/adapters/http/, adapter de persistência em src/adapters/db/
- E-mail normalizado (lowercase, trim) antes de validar e persistir
- Mensagem de erro de duplicidade não deve vazar se o e-mail já
  existe de forma que permita enumeração de usuários cadastrados

## Critérios de aceite
- [ ] Cadastro com dados válidos retorna 201 e o usuário criado
- [ ] E-mail inválido retorna 422 com mensagem clara
- [ ] E-mail duplicado retorna 409 sem vazar dados do usuário existente
- [ ] Suíte de testes cobre casos de sucesso e de erro (422, 409)

O humano revisa e faz um único ajuste: pede explicitamente a normalização do e-mail antes da validação, que o agente já havia incluído — mostrando que um bom arquiteto-back já antecipa requisitos óbvios, mas a revisão humana continua sendo o ponto de controle final antes de qualquer código ser escrito.

2. O plan.md

# plan.md — Implementação do cadastro de usuário

## Etapa 1 — Domínio
- src/domain/usuario.py — entidade Usuario (nome, email)
- src/domain/exceptions.py — EmailInvalidoError, EmailDuplicadoError

## Etapa 2 — Persistência
- src/adapters/db/models.py — modelo SQLAlchemy UsuarioModel
- src/adapters/db/repositorio_usuario.py — RepositorioUsuario
  (porta definida no domínio, adapter aqui implementa)

## Etapa 3 — Rota HTTP
- src/adapters/http/routes/usuarios.py — POST /usuarios
- schemas Pydantic v2 para request/response
- mapeamento de exceções de domínio para 422/409

## Etapa 4 — Testes (pytest)
- Unitários: validação de e-mail, normalização
- Integração: POST /usuarios com sucesso, e-mail inválido, e-mail duplicado
- Casos negativos: payload malformado, campos ausentes

## Etapa 5 — Auditoria de segurança
- Revisão específica do fluxo de cadastro (ver capítulo 9)

Note que o plano segue a mesma lógica de divisão em etapas menores discutida no capítulo 10 — cada etapa é um checkpoint natural para revisão humana, evitando que o agente implementador avance dezenas de arquivos sem supervisão intermediária.

3. Implementação e revisão de código

O dev-back segue o plano etapa por etapa. Um trecho da rota de cadastro, antes da revisão:

# ANTES (rejeitado pelo agente revisor)
# src/adapters/http/routes/usuarios.py
@router.post("/usuarios", status_code=201)
async def cadastrar_usuario(payload: dict, session: Session = Depends(get_session)):
    usuario = UsuarioModel(nome=payload["nome"], email=payload["email"])
    session.add(usuario)
    session.commit()
    return usuario

O revisor identifica dois problemas: (1) recebe payload: dict sem schema Pydantic, permitindo dados malformados chegarem direto ao banco; (2) não trata e-mail duplicado — a violação da constraint única do banco vaza como um IntegrityError não capturado (erro 500), em vez de um 409 controlado. O comentário do revisor: 

Revisor: Bloqueando este diff. Dois problemas:
1. payload: dict sem schema Pydantic — isso viola o requisito não
   funcional da spec (validação de nome e e-mail). Use o schema
   CriarUsuarioRequest já previsto no plano.
2. E-mail duplicado não é tratado explicitamente — hoje vaza como
   500 (IntegrityError do SQLAlchemy). Capture a exceção de domínio
   EmailDuplicadoError e mapeie para 409.

Depois da correção:

# DEPOIS (aprovado)
# src/adapters/http/routes/usuarios.py
@router.post("/usuarios", status_code=201, response_model=UsuarioResponse)
async def cadastrar_usuario(
    payload: CriarUsuarioRequest,
    repo: RepositorioUsuario = Depends(get_repositorio_usuario),
):
    try:
        usuario = Usuario(nome=payload.nome, email=payload.email)
        return repo.salvar(usuario)
    except EmailDuplicadoError:
        raise HTTPException(status_code=409, detail="Não foi possível concluir o cadastro")
    except EmailInvalidoError as exc:
        raise HTTPException(status_code=422, detail=str(exc))

Esse é exatamente o tipo de bug que "confiança cega" (capítulo 12) deixaria passar — o código antes da correção roda perfeitamente em qualquer teste manual feito com um e-mail novo. Só um revisor pensando em entrada não validada e em casos de erro reais (duplicidade) pega o problema.

4. QA e segurança

O qa escreve testes que replicam exatamente o cenário que o revisor havia previsto — cadastrar um e-mail já existente — e confirma que agora retorna 409 em vez de 500. O cyber-sec, por sua vez, focado especificamente no fluxo de cadastro, identifica que a mensagem de erro 409 original incluía o texto "e-mail já cadastrado", o que permite a um atacante enumerar quais e-mails já existem na base testando um a um — um desvio que passou despercebido na implementação e que o revisor de código (focado em lógica geral) não capturou, mas que é exatamente o tipo de detalhe que um agente especializado em segurança está calibrado para verificar.

Fluxo de trabalho recomendado

  1. Humano descreve o objetivo de negócio em linguagem natural, sem detalhar implementação.
  2. arquiteto-back gera spec.md; humano aprova ou ajusta.
  3. arquiteto-back gera plan.md dividido em etapas; humano aprova.
  4. dev-back codifica etapa por etapa, com checkpoints.
  5. revisor audita cada diff antes de seguir para a próxima etapa.
  6. qa escreve e roda testes cobrindo os critérios de aceite da spec.
  7. cyber-sec audita especificamente as áreas sensíveis (aqui, o cadastro de usuário).
  8. Humano faz o merge final e autoriza o deploy — a decisão de "vai para produção" nunca é do agente.

Dicas e boas práticas

Dica: Escreva os critérios de aceite da spec como testes em potencial — "e-mail duplicado retorna 409 sem vazar dados" é diretamente traduzível em um teste automatizado.
Dica: Deixe o cyber-sec atuar por último, depois que qa e revisor já aprovaram — ele deve auditar código já estável, não código em fluxo.
Dica: Cada "achado" de um agente (revisor, qa, cyber-sec) deve resultar em uma reexecução clara do dev-back — não em o humano corrigindo manualmente, o que quebraria a rastreabilidade do processo.

Erros comuns

Erro comum: Pular a etapa de revisão de código "porque o QA vai pegar os bugs depois" — revisor e qa verificam coisas diferentes (qualidade/lógica vs. comportamento observável), e vulnerabilidades como vazamento de informação numa mensagem de erro às vezes não aparecem em nenhum teste óbvio.
Erro comum: Deixar o cyber-sec revisar a spec inteira do zero, em vez de focar na área sensível (cadastro de usuário) — desperdiça tempo e dilui a atenção exatamente onde ela mais importa.

Exercício prático

Mão na massa: Reproduza este fluxo no seu user-api (ou um mini-projeto novo). Escreva você mesmo o pedido inicial em uma frase, peça ao arquiteto-back para gerar spec.md e plan.md, aprove/ajuste, e então implemente com pelo menos dois papéis distintos (dev-back + revisor). Observe quantos ajustes o revisor pede — isso é um indicador direto de quanto valor a etapa de revisão está agregando.

Resumo do capítulo

Este estudo de caso mostrou, ponta a ponta, como Spec-Driven Development e agentes especializados se combinam em um projeto real: o arquiteto-back gera spec.md e plan.md revisáveis pelo humano; o dev-back segue o plano em etapas; o revisor captura falhas de lógica e de segurança básica (como validação de entrada); o qa valida comportamento contra os critérios de aceite; e o cyber-sec audita especificamente a área mais sensível do sistema. O humano permanece no centro das decisões de aprovação em cada etapa — os agentes aceleram execução, mas não substituem julgamento.
