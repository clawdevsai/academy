13. Demonstração completa — um projeto construído com múltiplos agentes

Estudo de caso ponta a ponta: uma API de tarefas com autenticação, construída por cinco agentes especializados e um humano no comando.

🎯 Objetivos de aprendizagem

  • Ver, na prática, como Spec-Driven Development e multi-agentes se conectam em um fluxo real.
  • Entender a divisão de responsabilidades entre agente de planejamento, implementação, revisão, QA e segurança.
  • Analisar exemplos concretos de spec.md, plan.md e um diff revisado.
  • Identificar os pontos onde o humano precisa intervir e decidir.

Explicação conceitual

Este capítulo é diferente dos anteriores: em vez de explicar um conceito isoladamente, vamos narrar um projeto fictício do início ao fim, aplicando tudo o que foi visto no curso. O cenário: construir uma API de gerenciamento de tarefas (to-do list) com autenticação — simples o suficiente para caber em um capítulo, mas realista o suficiente para ter todas as arestas que aparecem em projetos de verdade: autenticação, validação, persistência, testes e uma superfície de ataque real (login, tokens, dados de usuário).

Vamos usar cinco papéis de agente, cada um com uma responsabilidade única — o mesmo princípio de especialização que vimos nos capítulos 5, 6 e 9:

Exemplos práticos

1. O pedido inicial e o spec.md

O humano começa com um pedido em linguagem natural, propositalmente não muito detalhado — é papel do agente planejador extrair as perguntas certas e formalizar: 

Você: Preciso de uma API REST de gerenciamento de tarefas (to-do list).
Cada usuário deve poder se cadastrar, fazer login e gerenciar
apenas suas próprias tarefas. Use um agente de planejamento
para gerar a spec antes de qualquer código.

O agente planejador devolve algo como:

# spec.md — API de Gerenciamento de Tarefas

## Objetivo
Fornecer uma API REST para que usuários autenticados criem,
listem, atualizem e removam suas próprias tarefas.

## Escopo
- Cadastro de usuário (email + senha)
- Login com emissão de JWT (access token) + refresh token
- CRUD de tarefas (título, descrição, status, prazo)
- Isolamento: um usuário nunca acessa tarefas de outro

## Fora de escopo (v1)
- Recuperação de senha por e-mail
- Compartilhamento de tarefas entre usuários
- Notificações

## Requisitos não funcionais
- Senhas armazenadas com hash (bcrypt, custo >= 12)
- Access token expira em 15 min; refresh token em 7 dias
- Rate limit no endpoint de login (máx. 5 tentativas / 15 min / IP)
- Todas as rotas de tarefas exigem autenticação

## Critérios de aceite
- [ ] Usuário A não consegue ler/editar tarefa de usuário B (testado)
- [ ] Senha nunca aparece em log ou resposta de API
- [ ] Suíte de testes cobre casos de sucesso e de erro (401, 403, 404)

O humano revisa e faz um único ajuste: pede explicitamente o rate limit no login, que o agente já havia incluído — mostrando que um bom agente de planejamento já antecipa requisitos de segurança óbvios, mas a revisão humana continua sendo o ponto de controle final antes de qualquer código ser escrito.

2. O plan.md

# plan.md — Implementação da API de Tarefas

## Etapa 1 — Modelagem de dados
- Tabela `users` (id, email único, password_hash, created_at)
- Tabela `tasks` (id, user_id FK, title, description, status, due_date)

## Etapa 2 — Autenticação
- POST /auth/register
- POST /auth/login (com rate limit)
- POST /auth/refresh
- Middleware de verificação de JWT

## Etapa 3 — CRUD de tarefas
- GET /tasks (só do usuário autenticado)
- POST /tasks
- PATCH /tasks/:id (valida ownership antes de alterar)
- DELETE /tasks/:id (valida ownership antes de remover)

## Etapa 4 — Testes
- Unitários: hashing, geração/validação de JWT
- Integração: fluxo completo de cadastro → login → CRUD
- Casos negativos: token inválido, acesso a tarefa de outro usuário

## Etapa 5 — Auditoria de segurança
- Revisão específica do fluxo de auth (ver capítulo 9)

Note que o plano segue a mesma lógica de divisão em etapas menores discutida no capítulo 10 — cada etapa é um checkpoint natural para revisão humana, evitando que o agente implementador avance dezenas de arquivos sem supervisão intermediária.

3. Implementação e revisão de código

O agente implementador segue o plano etapa por etapa. Um trecho da rota de atualização de tarefa, antes da revisão:

// ANTES (rejeitado pelo agente revisor)
app.patch('/tasks/:id', authMiddleware, async (req, res) => {
  const task = await db.task.update({
    where: { id: req.params.id },
    data: req.body
  });
  res.json(task);
});

O agente revisor identifica dois problemas: (1) não valida que a tarefa pertence ao usuário autenticado — qualquer usuário logado poderia editar a tarefa de outro só sabendo o ID; (2) usa req.body diretamente sem validação de schema, permitindo injeção de campos não previstos (ex.: sobrescrever user_id). O comentário do revisor: 

Revisor: Bloqueando este diff. Dois problemas:
1. Falta verificação de ownership — isso viola o critério de
   aceite da spec ("usuário A não acessa tarefa de B").
2. req.body sem validação de schema é uma superfície de ataque
   (mass assignment). Use o schema Zod já definido em
   schemas/task.js e adicione o filtro where: { id, userId }.

Depois da correção:

// DEPOIS (aprovado)
app.patch('/tasks/:id', authMiddleware, async (req, res) => {
  const data = updateTaskSchema.parse(req.body); // valida e filtra campos
  const task = await db.task.updateMany({
    where: { id: req.params.id, userId: req.user.id },
    data
  });
  if (task.count === 0) {
    return res.status(404).json({ error: 'Tarefa não encontrada' });
  }
  res.json({ ok: true });
});

Esse é exatamente o tipo de bug que "confiança cega" (capítulo 12) deixaria passar — o código antes da correção roda perfeitamente em qualquer teste manual feito pelo próprio usuário autenticado testando suas próprias tarefas. Só um revisor pensando em ownership e em outro usuário mal-intencionado pega o problema.

4. QA e segurança

O agente de QA escreve testes que replicam exatamente o cenário que o revisor havia previsto — usuário A tentando editar tarefa de usuário B — e confirma que agora retorna 404 (não vazando nem a existência da tarefa). O agente de segurança, por sua vez, focado especificamente no fluxo de autenticação, identifica que o access token estava configurado com expiração de 24 horas em vez dos 15 minutos definidos na spec — um desvio que passou despercebido na implementação e que o revisor de código (focado em lógica geral) não capturou, mas que é exatamente o tipo de detalhe que um agente especializado em segurança está calibrado para verificar.

Fluxo de trabalho recomendado

  1. Humano descreve o objetivo de negócio em linguagem natural, sem detalhar implementação.
  2. Agente planejador gera spec.md; humano aprova ou ajusta.
  3. Agente planejador gera plan.md dividido em etapas; humano aprova.
  4. Agente implementador codifica etapa por etapa, com checkpoints.
  5. Agente revisor audita cada diff antes de seguir para a próxima etapa.
  6. Agente de QA escreve e roda testes cobrindo os critérios de aceite da spec.
  7. Agente de segurança audita especificamente as áreas sensíveis (aqui, autenticação).
  8. Humano faz o merge final e autoriza o deploy — a decisão de "vai para produção" nunca é do agente.

Dicas e boas práticas

Dica: Escreva os critérios de aceite da spec como testes em potencial — "usuário A não acessa tarefa de B" é diretamente traduzível em um teste automatizado.
Dica: Deixe o agente de segurança atuar por último, depois que QA e revisor já aprovaram — ele deve auditar código já estável, não código em fluxo.
Dica: Cada "achado" de um agente (revisor, QA, segurança) deve resultar em uma reexecução clara do agente implementador — não em o humano corrigindo manualmente, o que quebraria a rastreabilidade do processo.

Erros comuns

Erro comum: Pular a etapa de revisão de código "porque o QA vai pegar os bugs depois" — revisor e QA verificam coisas diferentes (qualidade/lógica vs. comportamento observável), e vulnerabilidades como falta de ownership às vezes não aparecem em nenhum teste óbvio.
Erro comum: Deixar o agente de segurança revisar a spec inteira do zero, em vez de focar na área sensível (auth) — desperdiça tempo e dilui a atenção exatamente onde ela mais importa.

Exercício prático

Mão na massa: Reproduza este fluxo com um projeto seu (ou um mini-projeto novo). Escreva você mesmo o pedido inicial em uma frase, peça a um agente para gerar spec.md e plan.md, aprove/ajuste, e então implemente com pelo menos dois papéis distintos (implementador + revisor). Observe quantos ajustes o revisor pede — isso é um indicador direto de quanto valor a etapa de revisão está agregando.

Resumo do capítulo

Este estudo de caso mostrou, ponta a ponta, como Spec-Driven Development e agentes especializados se combinam em um projeto real: planejamento gera spec.md e plan.md revisáveis pelo humano; implementação segue o plano em etapas; revisão de código captura falhas de lógica e de segurança básica (como ownership); QA valida comportamento contra os critérios de aceite; e um agente de segurança dedicado audita especificamente a área mais sensível do sistema. O humano permanece no centro das decisões de aprovação em cada etapa — os agentes aceleram execução, mas não substituem julgamento.
