5. Como estruturar um workflow profissional com Claude Code

Saindo do "prompt avulso" para um processo repetível de trabalho em projetos reais.

🎯 Objetivos de aprendizagem

  • Entender por que "conversar com a IA" não é a mesma coisa que ter um workflow profissional.
  • Conhecer o padrão Explore → Plan → Code → Commit e quando aplicá-lo.
  • Estruturar um CLAUDE.md de projeto como memória persistente da equipe.
  • Configurar permissões e comandos customizados para reduzir fricção no dia a dia.
  • Saber quando parar e pedir revisão humana dentro do fluxo.

Explicação conceitual

Nos capítulos anteriores você viu as peças soltas: prompts, comandos, tools, MCP, skills e hooks. Um workflow profissional é a forma como você organiza essas peças em um processo repetível, de modo que qualquer pessoa do time — ou você mesmo, seis meses depois — consiga rodar a mesma tarefa com resultado previsível. Sem isso, cada sessão de Claude Code vira uma improvisação: às vezes funciona bem, às vezes o agente sai editando arquivos que não deveria, às vezes o contexto fica poluído e a qualidade cai.

O ponto de partida mental é um padrão que a própria Anthropic recomenda internamente: Explore → Plan → Code → Commit. A ideia é simples, mas contraintuitiva para quem vem do hábito de "pedir direto o código pronto":

  • Explore — peça ao Claude Code para ler os arquivos relevantes e resumir o que entendeu, sem escrever código ainda.
  • Plan — peça um plano de implementação (pode usar o modo de planejamento do Claude Code). Você revisa e ajusta antes de qualquer edição.
  • Code — só agora o agente implementa, seguindo o plano aprovado.
  • Commit — o agente (ou você) revisa o diff, roda testes e faz o commit com uma mensagem clara.

Esse padrão evita o maior risco de produtividade com agentes de IA: deixá-los "correr solto" gerando muito código antes de qualquer checkpoint humano. Quanto maior a tarefa, mais caro fica descobrir tarde que a direção estava errada.

Exemplos práticos

O arquivo CLAUDE.md na raiz do projeto é lido automaticamente pelo Claude Code no início de cada sessão. Ele funciona como a "memória de longo prazo" do projeto — convenções, comandos de build, arquitetura, coisas a evitar. 

# CLAUDE.md

## Stack
- Next.js 14 (App Router) + TypeScript + Prisma + PostgreSQL

## Comandos úteis
- `npm run dev` — sobe o ambiente local
- `npm run test` — roda a suíte de testes (Vitest)
- `npm run lint` — ESLint + Prettier

## Convenções
- Componentes em `src/components`, sempre com testes ao lado (`Componente.test.tsx`)
- Nunca alterar arquivos em `src/generated/**` (gerados pelo Prisma)
- Commits seguem Conventional Commits (feat:, fix:, chore:...)

## O que NÃO fazer
- Não fazer `git push --force` na branch main
- Não instalar dependências novas sem perguntar antes

Comandos customizados (slash commands) reduzem prompts repetitivos. Eles ficam em .claude/commands/: 

# .claude/commands/revisar-pr.md
Revise o diff atual como um engenheiro sênior faria em um PR review.
Aponte: bugs, problemas de segurança, código duplicado e testes faltando.
Não corrija nada ainda — apenas liste os pontos encontrados.

Permissões ficam em .claude/settings.json, evitando que você precise aprovar a mesma ação repetidamente ou, pior, aprove ações perigosas sem pensar: 

{
  "permissions": {
    "allow": ["Bash(npm run test:*)", "Bash(npm run lint:*)"],
    "deny": ["Bash(git push --force:*)", "Bash(rm -rf:*)"]
  }
}

Fluxo de trabalho recomendado

  1. Antes de começar, garanta que o CLAUDE.md está atualizado com o estado real do projeto.
  2. Descreva a tarefa e peça explicitamente para o agente explorar antes de agir.
  3. Peça um plano por escrito. Leia com atenção — é o ponto mais barato para corrigir rumo.
  4. Aprove ou ajuste o plano. Só então libere a implementação.
  5. Depois do código gerado, rode testes e lint automaticamente (via hooks ou comando manual).
  6. Revise o diff final antes do commit — nunca faça commit "às cegas" de código gerado por IA.

Dica: use o modo de planejamento (plan mode) do Claude Code para tarefas médias ou grandes — ele impede edições até que você aprove explicitamente o plano.
Dica: mantenha o CLAUDE.md curto e vivo. Um arquivo de 3 páginas que ninguém atualiza é pior do que um de 20 linhas sempre correto.
Dica: versione .claude/commands/ e .claude/settings.json junto com o código — assim todo o time herda o mesmo workflow.

Erros comuns

Erro comum: pular a fase de exploração e plano em tarefas grandes "para ir mais rápido". Resultado: o agente implementa na direção errada e você perde mais tempo desfazendo do que teria gasto planejando.
Erro comum: aprovar permissões amplas demais (ex: liberar todo comando Bash) só para parar de ser interrompido. Isso remove a rede de segurança justamente quando ela é mais necessária — prefira listas específicas de allow/deny.
Erro comum: tratar o commit gerado pelo agente como definitivo sem ler o diff. Mesmo com um bom workflow, revisão humana final continua sendo sua responsabilidade.
Mão na massa: escolha um projeto real (ou crie um pequeno). Crie um CLAUDE.md com stack, comandos e convenções. Depois peça ao Claude Code uma tarefa pequena seguindo explicitamente as quatro fases: "explore o projeto", "monte um plano", "implemente conforme o plano" e "prepare o commit". Compare o resultado com o que aconteceria se você tivesse pedido tudo em um único prompt.

Resumo do capítulo

Um workflow profissional transforma o Claude Code de "assistente de prompt único" em um processo repetível e auditável. O padrão Explore → Plan → Code → Commit cria checkpoints de revisão humana antes que erros fiquem caros. O CLAUDE.md funciona como memória persistente do projeto, enquanto comandos customizados e permissões em settings.json reduzem fricção sem abrir mão de segurança. Esse workflow é a base para os próximos capítulos, que tratam de dividir esse processo entre múltiplos agentes especializados.
