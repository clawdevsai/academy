15. Workflows Dinâmicos — Orquestrando Múltiplos Agentes em Escala

Quando uma conversa não basta: fazer Claude escrever um script que coordena dezenas de agentes sozinho.

🎯 Objetivos de aprendizagem

  • Entender a diferença entre subagente, skill e workflow — quem mantém o plano em cada caso.
  • Saber quando vale a pena trocar uma conversa por um workflow (e quando é desperdício de tokens).
  • Conhecer os 6 padrões compostos que cobrem quase todo caso de uso de orquestração em escala.
  • Aplicar esses 6 padrões usando os agentes especializados já criados no projeto (arquiteto, engenheiro, revisor, QA, segurança, DevOps).

Explicação conceitual

Até aqui, todo agente que você criou era acionado dentro de uma conversa: você (ou o Claude principal) decidia, turno por turno, quando chamar o lw-code-reviewer ou o lw-qa-engineer, e o resultado de cada um voltava para o contexto da conversa. Isso funciona bem até a tarefa ficar grande demais para caber num vaivém de turnos — uma auditoria em 50 módulos, uma migração de 500 arquivos, uma pesquisa que precisa cruzar dezenas de fontes.

Um workflow dinâmico é a resposta a esse problema: em vez de o Claude principal decidir a cada turno o que fazer a seguir, ele escreve um script (JavaScript) que descreve toda a orquestração de uma vez — quantos agentes rodam, em que ordem, o que cada um recebe, como os resultados se combinam. Um runtime executa esse script em segundo plano, sua sessão continua livre, e só a resposta final chega ao contexto da conversa. Os resultados intermediários ficam em variáveis do script, não no contexto do Claude.

Limites que valem saber antes de usar

  • Até 16 agentes simultâneos (menos em máquinas com poucos núcleos).
  • Até 1.000 agentes no total por execução — trava loops descontrolados.
  • Nenhuma entrada do usuário durante a execução: se você precisa aprovar entre etapas, rode cada etapa como seu próprio workflow.
  • Os subagentes de um workflow sempre rodam em modo acceptEdits — edições de arquivo são aprovadas automaticamente, independente do modo da sua sessão.

Por que workflows existem: as 3 falhas que eles corrigem

Quanto mais tempo o Claude trabalha numa tarefa complexa dentro de uma única janela de contexto, mais suscetível fica a três falhas específicas — e workflows existem justamente para combatê-las orquestrando subagentes separados, cada um com contexto próprio e objetivo isolado:

  • Preguiça agencial (agentic laziness): parar antes de terminar uma tarefa longa e declarar "concluído" após progresso parcial — por exemplo, resolver 35 de 50 itens de uma revisão de segurança.
  • Viés autopreferencial (self-preferential bias): tendência de preferir o próprio resultado, especialmente quando pedido para verificar ou julgar o próprio trabalho — é exatamente o que o padrão de Verificação Adversarial existe para neutralizar.
  • Deriva de objetivo (goal drift): perda gradual de fidelidade ao objetivo original ao longo de muitos turnos, principalmente após compactação de contexto — cada resumo é sujeito a perder detalhes como uma restrição de "não faça X".

Os 6 padrões de workflow

Praticamente todo caso de uso de orquestração em escala é uma combinação destes seis padrões, documentados pela própria Anthropic. Vale aprender os nomes: pedir um workflow citando o padrão certo dá resultados mais previsíveis do que descrever a tarefa solta.

Os 6 padrões de workflow — adaptado de "A harness for every task" (Anthropic, jun/2026)

1. Classify-and-Act (classificar e agir)

Um agente classificador olha cada item e decide o tipo, roteando para o especialista certo.

Cenário no projeto: uma lista de issues abertas precisa ser distribuída entre os agentes certos sem que você leia uma por uma.

Rode um workflow para classificar cada issue aberta no repositório: bugs de
implementação vão para o lw-backend-engineer, decisões de arquitetura para o
lw-backend-architect, e suspeitas de vulnerabilidade para o lw-cybersecurity-engineer.
Padrão relacionado — quarentena: em triagens que processam conteúdo não confiável (issues abertas por qualquer pessoa, tickets externos), não deixe o classificador — que lê esse conteúdo bruto — executar ações de alto privilégio diretamente. A ação (corrigir, mergear, escalar) fica a cargo de um agente separado que só recebe a classificação já feita.

2. Fan-Out / Synthesize (espalhar e sintetizar)

Divide o trabalho em unidades independentes, roda um agente por unidade em contexto limpo (sem contaminação cruzada), e um passo final sintetiza tudo.

Cenário no projeto: auditar todos os módulos do projeto sem revisar módulo por módulo numa conversa só.

Rode um workflow para revisar, com o lw-code-reviewer, cada módulo dentro de
src/ em paralelo, e sintetize um único relatório consolidado por severidade.

3. Adversarial Verification (verificação adversarial)

Para cada resultado que um agente produz, um segundo agente é acionado só para desafiar esse resultado contra critérios objetivos — contrapondo a tendência do modelo de confiar demais no próprio trabalho.

Cenário no projeto: confirmar que uma correção de segurança realmente resolve o problema, não só esconde o sintoma.

Depois que o lw-backend-engineer corrigir a vulnerabilidade em auth.py, rode um
workflow onde o lw-cybersecurity-engineer verifica adversarialmente se a correção
realmente elimina o risco, antes de aceitar como resolvido.

4. Generate-and-Filter (gerar e filtrar)

Gera muitos candidatos, filtra por um critério ou verificação, remove quase-duplicados e devolve só os sobreviventes de maior qualidade.

Cenário no projeto: explorar várias alternativas de design antes de se comprometer com uma.

Rode um workflow onde o lw-backend-architect gera 5 alternativas de design para
o módulo de pagamentos, e filtre por segurança e simplicidade até sobrar só as
2 melhores.

5. Tournament (torneio)

Em vez de dividir o trabalho, os agentes competem nele: cada um tenta a mesma tarefa com uma abordagem diferente, e um agente juiz avalia os confrontos par a par até restar um vencedor.

Cenário no projeto: comparar implementações candidatas de uma função sensível a performance.

Rode um workflow onde três implementações candidatas da função de cache competem
em um torneio, com o lw-qa-engineer e o lw-backend-architect julgando cada
confronto par a par por corretude e performance.

6. Loop Until Done (repetir até terminar)

Para tarefas com quantidade de trabalho desconhecida, o workflow repete a geração de agentes até uma condição real de parada ser atingida (sem novos achados, sem mais erros), em vez de um número fixo de passagens.

Cenário no projeto: é exatamente a lógica por trás do gate de pré-push do Dia 3 — só que rodando sozinho, sem você orquestrar manualmente.

Rode um workflow onde o lw-qa-engineer roda a suíte de testes e, se falhar,
aciona o lw-backend-engineer para corrigir — repetindo até todos os testes
passarem.

Exemplos práticos

Para rodar o workflow embutido de pesquisa profunda:

/deep-research O que mudou no modelo de permissões do Node.js entre v20 e v22?

Para pedir um workflow sob medida, basta incluir a palavra workflow no prompt — o Claude Code destaca a palavra e escreve um script em vez de trabalhar turno por turno:

Rode um workflow para auditar todos os endpoints em src/routes/
procurando checagens de autorização ausentes, usando o lw-cybersecurity-engineer
por módulo e um passo de verificação adversarial em cada achado antes de reportar.

Antes de a execução começar, o Claude Code mostra as fases planejadas e pergunta: Sim, execute / Ver script bruto (ou Ctrl+G para abrir no editor) / Não. Acompanhe o progresso a qualquer momento com:

/workflows

Se a execução fizer o que você queria, selecione-a em /workflows e aperte s para salvar como comando reutilizável — em .claude/workflows/ (compartilhado com o time) ou ~/.claude/workflows/ (só seu). A partir daí ele vira um /nome-do-workflow disponível em qualquer sessão.

Dicas e boas práticas

Dica: use workflows para tarefas que realmente precisam de paralelismo, verificação cruzada ou escala — auditorias grandes, migrações, decisões que merecem múltiplos ângulos. Para um bug de duas linhas, um workflow é overkill e caro.
Dica: sempre dê uma olhada no "Ver script bruto" (ou Ctrl+G) antes de aprovar uma execução grande — é um script de verdade, revisável como qualquer código.
Dica: adicione à lista de permissões os comandos que os agentes vão precisar antes de rodar um workflow longo — durante a execução, um comando fora da lista ainda pode gerar um prompt de confirmação e travar o fluxo.
Dica: workflows aceitam orçamento de tokens explícito no próprio prompt — algo como "use no máximo 10 mil tokens" já limita o teto de gasto da execução. Use isso em demos ao vivo para evitar surpresas de custo.
Dica: para tarefas recorrentes (triagem, pesquisa, verificação), combine o workflow salvo com /loop (roda em intervalos regulares) e /goal (define uma condição real de conclusão, em vez de parar num ponto superficial).
Quando NÃO usar workflows: a maioria das tarefas de programação do dia a dia não precisa de mais poder computacional — um bug de duas linhas não precisa de um painel de 5 revisores. Pergunte-se: essa tarefa realmente se beneficia de paralelismo, verificação cruzada ou múltiplos ângulos independentes? Se a resposta for não, uma conversa normal (ou um subagente único) resolve com muito menos tokens.

Erros comuns

Erro comum: usar um workflow para uma tarefa trivial "porque é mais impressionante". O custo em tokens de um workflow com vários agentes é significativamente maior que uma conversa normal — reserve para o que realmente precisa de escala.
Erro comum: esperar poder interromper e dar instruções no meio da execução. Não há entrada do usuário durante um workflow rodando — se você precisa de um checkpoint de aprovação humana no meio, divida a tarefa em vários workflows menores.
Erro comum: esquecer que os agentes de um workflow rodam sempre em acceptEdits — todas as edições de arquivo são aprovadas automaticamente. Se a lista de ferramentas permitidas dos seus agentes especializados for permissiva demais, um workflow pode aplicar muitas mudanças de uma vez sem revisão intermediária.

Outros usos que valem conhecer

A própria Anthropic documenta workflows aplicados bem além de código: o Bun foi reescrito de Zig para Rust usando workflows (um agente por ponto de falha, revisão adversarial, merge); times usam workflows para minerar canais de incidentes no Slack em busca de causas-raiz recorrentes sem ticket aberto; para rankear grandes listas (ex: 80 currículos, milhares de tickets) por comparação par-a-par em vez de nota absoluta — muito mais confiável do que pedir para o modelo pontuar 1000+ itens de uma vez; e até para investigação de causa-raiz fora de código, gerando hipóteses a partir de evidências desconexas (logs, dados, arquivos) e as confrontando com um painel de verificadores.

Exercício prático

Mão na massa: no seu projeto Python, peça um workflow usando o padrão Fan-Out/Synthesize: "Rode um workflow para revisar, com o lw-code-reviewer, cada módulo dentro de src/ em paralelo, e sintetize um único relatório consolidado por severidade." Depois, abra /workflows e observe as fases, os agentes e o consumo de tokens de cada um.

Resumo do capítulo

Workflows dinâmicos movem o plano de orquestração para um script que o runtime executa, liberando sua sessão e permitindo dezenas a centenas de agentes por execução — algo que uma conversa turno por turno não sustenta. Os seis padrões (Classify-and-Act, Fan-Out/Synthesize, Adversarial Verification, Generate-and-Filter, Loop Until Done e Tournament) cobrem quase todo caso de orquestração em escala, e cada um deles pode ser montado diretamente sobre os agentes especializados que vocês já criaram no projeto. Use workflows quando a tarefa exigir paralelismo ou verificação cruzada real — não como recurso padrão para qualquer tarefa.
