7. Como criar e organizar agentes personalizados

A mecânica: arquivo, frontmatter, prompt, tools, testagem — passo a passo de criar um subagente.

🎯 Objetivos de aprendizagem

  • Saber onde e como criar um arquivo de subagente em Markdown.
  • Entender cada campo do frontmatter: name, description, tools, model.
  • Escrever um system prompt eficaz para um agente especializado.
  • Diferenciar agentes de projeto (.claude/agents/) de agentes de usuário (~/.claude/agents/).
  • Testar e iterar sobre um agente customizado antes de confiar nele em produção.

Explicação conceitual

O capítulo anterior tratou de quando e por que dividir trabalho entre agentes. Este capítulo é sobre a mecânica: como o Claude Code efetivamente sabe que um subagente existe e quando usá-lo.

Um subagente é, na prática, um arquivo .md dentro de uma pasta agents. Ele tem duas partes: um bloco de frontmatter (metadados em YAML, entre ---) e o corpo, que é o system prompt do agente — as instruções que definem como ele se comporta.

CampoObrigatórioFunção
nameSimIdentificador único do agente, usado para invocação explícita.
descriptionSimDescrição em linguagem natural de quando usar este agente. É o que o Claude Code lê para decidir se delega a tarefa a ele automaticamente.
toolsNãoLista de tools que o agente pode usar. Se omitido, herda todas as tools disponíveis — quase sempre vale restringir.
modelNãoQual modelo usar (ex: um modelo menor/mais rápido para tarefas simples e repetitivas, um mais forte para raciocínio complexo).

A description é o campo mais importante e o mais subestimado. É com base nela que o agente principal decide, sozinho, se delega uma tarefa ao subagente — sem você precisar chamar o nome dele explicitamente. Uma descrição vaga ("agente para ajudar com código") faz o Claude Code nunca escolher esse agente, ou escolhê-lo na hora errada. Uma descrição específica ("use este agente sempre que precisar revisar um diff em busca de bugs de segurança antes de um merge") aciona o agente certo, na hora certa.

Existem dois escopos de armazenamento: .claude/agents/ dentro do repositório (agente de projeto, versionado e compartilhado com o time) e ~/.claude/agents/ na pasta do usuário (agente pessoal, disponível em todos os seus projetos, não compartilhado). Use o escopo de projeto para papéis ligados às convenções específicas daquele repositório, e o escopo de usuário para hábitos pessoais que você quer levar para qualquer projeto que abrir.

flowchart TD U[Usuário descreve tarefa] --> C{Claude Code avalia} C -->|combina com description de algum agente| D[Delega ao subagente] C -->|nenhum match claro| E[Resolve na conversa principal] D --> F["Subagente executa com suas tools e seu system prompt"] F --> G[Devolve resultado resumido ao agente principal]

Exemplos práticos

Exemplo completo de um agente de revisão de código, salvo em .claude/agents/revisor-codigo.md:

---
name: revisor-codigo
description: >
  Use este agente sempre que houver um diff de código pronto para revisão,
  antes de qualquer commit ou abertura de PR. Ele NÃO deve escrever ou
  corrigir código — apenas apontar problemas.
tools: Read, Grep, Glob
model: claude-sonnet-4.5
---

Você é um revisor de código sênior, cético e detalhista.

Ao receber um diff ou um conjunto de arquivos alterados:
1. Verifique lógica incorreta, edge cases não tratados e nomes pouco claros.
2. Verifique se há testes cobrindo a mudança; se não houver, aponte isso.
3. Verifique padrões do projeto definidos em CLAUDE.md.
4. Não sugira reescrever tudo — priorize os 3-5 problemas mais importantes.
5. Sempre termine com um veredito claro: "aprovado", "aprovado com ressalvas"
   ou "não aprovado", e o motivo.

Nunca edite arquivos. Seu papel é apenas revisar e reportar.

Outro exemplo, um agente de QA restrito a rodar testes (sem acesso a edição de código de produção):

---
name: qa-tester
description: >
  Use este agente para escrever e executar testes automatizados depois que
  uma feature for implementada. Também usado para validar se um bug foi
  corrigido de fato.
tools: Read, Write, Bash, Grep
---

Você é um engenheiro de QA. Seu foco é cobertura de testes e casos de borda.

Ao ser acionado:
1. Leia o código relevante para entender o comportamento esperado.
2. Escreva ou atualize testes automatizados (unitários e, quando fizer
   sentido, de integração).
3. Rode a suíte de testes via `npm run test` e reporte falhas com clareza.
4. Nunca altere código de produção — se encontrar um bug, reporte-o em vez
   de corrigi-lo.

Para testar um agente novo, você pode invocá-lo explicitamente pelo nome:

Use o agente revisor-codigo para revisar as mudanças que acabei de fazer
em src/services/payment.ts

Fluxo de trabalho recomendado

  1. Identifique um papel recorrente e bem delimitado no seu processo (ex: "sempre reviso segurança antes de mergear").
  2. Crie o arquivo em .claude/agents/nome-do-agente.md com frontmatter completo.
  3. Escreva a description pensando em "quando o agente principal deveria escolher isso sozinho".
  4. Restrinja tools ao mínimo necessário para o papel (um revisor não precisa de Write, por exemplo).
  5. Teste invocando o agente explicitamente algumas vezes antes de confiar na delegação automática.
  6. Depois de validado, teste em um cenário real sem citar o nome do agente — veja se o Claude Code o escolhe sozinho.
  7. Refine a description se a delegação automática não estiver acontecendo como esperado.
Dica: escreva a description na terceira pessoa e comece com "Use este agente quando..." — isso ajuda o modelo principal a fazer o matching corretamente.
Dica: restringir tools não é só sobre segurança — também mantém o agente focado. Um agente com acesso a tudo tende a "sair do papel" com mais frequência.
Dica: para tarefas simples e repetitivas (ex: formatar mensagens de commit), vale configurar um model menor/mais barato; reserve o modelo mais forte para agentes que precisam raciocinar sobre arquitetura.

Erros comuns

Erro comum: escrever uma description genérica demais ("ajuda com o código"). O agente principal nunca vai conseguir decidir quando delegar, e o subagente customizado vira letra morta.
Erro comum: dar todas as tools por padrão "para não dar erro depois". Isso aumenta a superfície de risco (um agente de revisão que pode editar arquivos, por exemplo) e reduz o foco do agente no seu papel.
Erro comum: criar um agente e nunca testá-lo isoladamente. Sem testar a invocação explícita primeiro, fica difícil saber se um resultado ruim veio do prompt do agente ou de uma delegação mal feita.
Mão na massa: crie um agente .claude/agents/documentador.md cuja única responsabilidade é gerar ou atualizar comentários e documentação de funções (sem tocar em lógica). Defina tools: Read, Edit e escreva uma description específica. Teste invocando-o explicitamente e depois, em uma tarefa nova, veja se o Claude Code o aciona sozinho ao perceber que falta documentação.

Resumo do capítulo

Um subagente é um arquivo Markdown com frontmatter (name, description, tools, model) e um system prompt no corpo. A description é o campo que determina se a delegação automática funciona bem — ela deve ser específica sobre quando usar o agente. Agentes de projeto vivem em .claude/agents/ e são compartilhados com o time; agentes de usuário vivem em ~/.claude/agents/ e seguem você entre projetos. Restringir tools ao mínimo necessário mantém o agente seguro e focado. O próximo capítulo mostra como formalizar ainda mais o processo de trabalho com Spec-Driven Development.
