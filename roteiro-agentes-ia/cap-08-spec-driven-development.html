8. Spec-Driven Development com Claude Code

Especificar e planejar antes de codificar: revisão humana concentrada em poucos pontos de alto valor.

🎯 Objetivos de aprendizagem

  • Entender o que é Spec-Driven Development (SDD) e o problema que ele resolve.
  • Conhecer os três a quatro documentos do fluxo: Specify, Plan, Tasks/Implement.
  • Saber onde entram os checkpoints de revisão humana e por que eles importam.
  • Conhecer ferramentas do ecossistema: GitHub Spec Kit, BMAD-METHOD, Superpowers.
  • Comparar SDD com "vibe coding" e prompting ad-hoc, e saber quando cada abordagem vale a pena.

Explicação conceitual

Nos capítulos 5 a 7 você viu como estruturar workflow, dividir agentes e criar subagentes. Spec-Driven Development (SDD, ou "desenvolvimento guiado por especificação") é um passo além: em vez de revisar o trabalho do agente dezenas de vezes durante a implementação, você concentra a revisão humana em poucos documentos escritos antes de qualquer código ser gerado.

A lógica é econômica: quanto mais cedo você pega um erro de entendimento, mais barato é corrigi-lo. Corrigir uma frase ambígua numa especificação custa segundos. Corrigir a mesma ambiguidade depois que ela virou 400 linhas de código, testes e integração custa horas. SDD empurra a atenção humana para o início do processo, onde ela é mais barata.

O fluxo, na prática, tem de três a quatro etapas:

  • Specify — uma especificação curta do que a mudança deve fazer, do ponto de vista de comportamento observável (não de implementação). Costuma incluir uma fase de perguntas de esclarecimento, em que o próprio agente pergunta o que está ambíguo antes de prosseguir.
  • Plan — transforma a especificação em um plano técnico: define "o que significa estar pronto" (critério de aceite), mapeia os arquivos afetados, descreve a jornada de uso e quebra o trabalho em uma lista numerada de tarefas.
  • Tasks / Implement — o Claude Code implementa tarefa por tarefa a partir da lista, com revisão humana entre etapas (não apenas no final).

Esse modelo é formalizado por ferramentas específicas do ecossistema: GitHub Spec Kit (framework oficial do GitHub para SDD, com comandos como /specify, /plan e /tasks), BMAD-METHOD (metodologia orientada a agentes com papéis definidos ao longo do ciclo de especificação) e Superpowers (conjunto de skills que reforça esse fluxo dentro do Claude Code). Você não precisa adotar uma ferramenta específica para praticar SDD — os três documentos podem ser simples arquivos Markdown versionados junto do código.

Exemplos práticos

Esqueleto de um documento de especificação, specs/checkout-parcelado/spec.md:

# Spec: Checkout parcelado

## Objetivo
Permitir que o cliente parcele o pagamento em até 12x no checkout,
respeitando o valor mínimo de parcela de R$ 20.

## Comportamento esperado
- O campo de parcelas só aparece quando o valor total ultrapassa R$ 40.
- O número máximo de parcelas é calculado dinamicamente pelo valor mínimo.
- Juros: sem juros até 6x; 1,99% a.m. de 7x a 12x.

## Fora de escopo
- Parcelamento no boleto (fica para outra iteração).

## Perguntas em aberto
- O limite de 12x vale para todos os métodos de pagamento ou só cartão?

Esqueleto do plano gerado a partir da spec aprovada, specs/checkout-parcelado/plan.md:

# Plano: Checkout parcelado

## Definição de pronto
- Testes cobrindo cálculo de parcelas e regra de valor mínimo
- UI atualizada com seletor de parcelas
- Sem regressão no fluxo de pagamento à vista

## Arquivos afetados
- src/checkout/PaymentForm.tsx
- src/checkout/calculateInstallments.ts (novo)
- src/checkout/PaymentForm.test.tsx

## Tarefas
1. Criar função calculateInstallments com testes unitários
2. Integrar seletor de parcelas na UI condicionalmente
3. Atualizar testes de integração do checkout
4. Validar manualmente os cenários de 1x, 6x e 12x

Com o GitHub Spec Kit instalado, o fluxo vira comandos dentro do Claude Code:

/specify Quero permitir parcelamento no checkout em até 12x
/plan
/tasks

Fluxo de trabalho recomendado

  1. Escreva ou peça ao Claude Code para gerar a spec a partir da sua descrição inicial da necessidade.
  2. Responda às perguntas de esclarecimento que o agente levantar — não pule essa fase, é onde ambiguidades aparecem.
  3. Revise a spec como humano: ela descreve comportamento, não implementação? Está completa?
  4. Peça o plano a partir da spec aprovada. Revise arquivos afetados e a lista de tarefas.
  5. Autorize a implementação tarefa por tarefa — revise entre tarefas, especialmente nas primeiras, que definem padrões para as seguintes.
  6. Ao final, valide contra a definição de pronto registrada no plano, não apenas "parece que funciona".

Dica: guarde spec e plano como arquivos versionados junto do código (não só na conversa) — eles viram documentação viva da decisão tomada.
Dica: use SDD para features médias e grandes; para um ajuste de uma linha, o overhead de escrever spec e plano não compensa.
Dica: combine SDD com os agentes especializados do capítulo 7 — por exemplo, um agente "arquiteto" pode ser o responsável por gerar o plano a partir da spec.

Erros comuns

Erro comum: escrever a spec já pensando em implementação ("usar Redis para cache") em vez de comportamento ("a resposta deve ser servida em menos de 200ms mesmo sob carga"). Isso trava decisões técnicas cedo demais, antes do plano avaliar as opções.
Erro comum: aprovar a spec sem responder às perguntas de esclarecimento, só para "andar mais rápido". A ambiguidade não desaparece — ela só reaparece mais tarde, dentro do código já escrito.
Erro comum: revisar apenas no final da implementação, tratando as "tarefas" do plano como uma lista para o agente rodar sem parar. Isso recria o mesmo problema que o SDD tenta resolver — revisão tardia e cara.
Mão na massa: escolha uma feature média do seu backlog. Peça ao Claude Code para gerar uma spec com uma fase de perguntas de esclarecimento; responda às perguntas; aprove a spec; peça o plano; revise arquivos afetados e tarefas; só então autorize a implementação tarefa por tarefa. Ao final, compare quantas vezes você precisou intervir comparado a fazer a mesma feature em um prompt único.

Resumo do capítulo

Spec-Driven Development concentra a revisão humana em documentos escritos antes da implementação — Specify, Plan e Tasks/Implement — em vez de espalhá-la (ou eliminá-la) durante a codificação. A spec descreve comportamento esperado e passa por uma fase de esclarecimento; o plano traduz isso em arquivos afetados, definição de pronto e tarefas numeradas; a implementação segue tarefa por tarefa com revisão entre etapas. Ferramentas como GitHub Spec Kit, BMAD-METHOD e Superpowers formalizam esse fluxo, mas a prática pode começar com simples arquivos Markdown versionados. O próximo capítulo aplica tudo isso — agentes especializados e processo estruturado — aos papéis de revisão, QA, segurança e DevOps.
