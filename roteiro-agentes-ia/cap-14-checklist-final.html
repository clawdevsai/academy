14. Checklist final de boas práticas

Uma referência rápida e standalone para consultar antes — e durante — qualquer projeto real com agentes de IA.

🎯 Objetivos de aprendizagem

  • Ter uma referência única, organizada por categoria, para consultar rapidamente em projetos reais.
  • Revisar de forma consolidada os pontos-chave de todo o curso.
  • Usar o checklist como gate de qualidade antes de iniciar ou concluir uma tarefa com agentes.

Explicação conceitual

Este capítulo não introduz conceitos novos — ele consolida tudo o que foi visto nos 13 capítulos anteriores em um formato de consulta rápida. A ideia é que você volte a esta página sempre que estiver iniciando um projeto novo, revisando um PR gerado por IA, ou decidindo se pode confiar em um agente para uma tarefa mais arriscada. Os itens estão organizados em seis categorias que cobrem o ciclo de vida completo de um projeto com agentes de IA.

Checklist consolidado

1. Planejamento

  • O objetivo da tarefa está descrito com critério de aceite mensurável, não apenas "melhora isso"?
  • Para tarefas médias/grandes, existe uma spec.md aprovada antes de qualquer código ser escrito?
  • O nível de processo (direto / spec curta / SDD completo) é proporcional ao risco real da tarefa?
  • Decisões arquiteturais relevantes foram documentadas (ADR ou seção equivalente no CLAUDE.md)?
  • Tarefas grandes foram divididas em etapas com checkpoints de revisão intermediários?

2. Configuração de agentes

  • Existe um CLAUDE.md atualizado com stack, convenções e comandos do projeto?
  • Agentes customizados/especializados estão versionados junto com o código (não só na máquina local)?
  • Cada agente tem um papel único e bem definido (implementador, revisor, QA, segurança) em vez de um agente genérico fazendo tudo?
  • Você começou com um agente/uma tarefa antes de escalar para orquestração multi-agente?
  • Prompts recorrentes e eficazes foram salvos/documentados para reuso, em vez de reescritos do zero toda vez?

3. Segurança

  • Hooks e guardrails de segurança estão ativos — e nunca desligados "temporariamente" sob pressão de prazo?
  • Áreas sensíveis (autenticação, pagamento, dados pessoais) passaram por um agente de segurança dedicado?
  • Segredos e credenciais nunca são colados diretamente em prompts ou commitados no repositório?
  • Existe verificação explícita de ownership/autorização em toda rota que manipula dados de usuário?
  • O agente tem permissões mínimas necessárias (princípio do menor privilégio) para executar a tarefa?

4. Qualidade

  • Todo diff gerado por IA é revisado com o mesmo rigor de um PR de um colega humano?
  • Existe suíte de testes automatizados cobrindo os critérios de aceite da spec?
  • Casos negativos (erro, acesso não autorizado, dados inválidos) estão cobertos, não só o caminho feliz?
  • Um agente de QA roda a suíte completa antes de considerar a tarefa concluída?
  • Você resiste à tentação de aceitar output só porque "os testes passaram", sabendo que testes cobrem apenas o que foi antecipado?

5. Contexto / Tokens

  • Buscas amplas, leitura de logs extensos ou exploração de código foram delegadas a subagentes isolados?
  • Informação repetida em múltiplos prompts foi movida para o CLAUDE.md em vez de copiada toda vez?
  • Você evita colar arquivos inteiros quando um trecho específico resolveria?
  • Conversas longas e não relacionadas foram encerradas/resumidas em vez de acumular contexto irrelevante?
  • Tarefas grandes foram divididas para não esgotar a janela de contexto no meio da execução?

6. Deploy / DevOps

  • Existe um commit limpo no git antes de qualquer tarefa arriscada — sua rede de segurança mínima?
  • A decisão final de merge e deploy é sempre humana, nunca automática pelo agente?
  • Pipelines de CI rodam a mesma suíte de testes usada localmente pelo agente de QA?
  • Um agente de DevOps/infra foi consultado para mudanças de configuração de produção, e não apenas de código de aplicação?
  • Existe plano de rollback claro caso a mudança gerada por agente cause um incidente em produção?

Exemplos práticos

Use este checklist como um "gate" em dois momentos específicos do seu fluxo de trabalho: antes de iniciar uma tarefa (categorias 1 e 2 — planejamento e configuração) e antes de fazer merge (categorias 3, 4, 5 e 6 — segurança, qualidade, contexto e deploy). Por exemplo, ao revisar um PR gerado por um agente implementador, percorra rapidamente os itens de "Segurança" e "Qualidade" como uma auditoria de dois minutos antes de aprovar. 

Antes de iniciar:
  [x] Objetivo com critério de aceite claro
  [x] CLAUDE.md atualizado
  [x] Nível de processo escolhido (spec curta, neste caso)

Antes do merge:
  [x] Diff revisado como PR humano
  [x] Suíte de testes verde, incluindo casos negativos
  [x] Nenhum guardrail de segurança desligado
  [x] Commit limpo anterior existe (rollback possível)

Fluxo de trabalho recomendado

  1. Abra este capítulo (ou salve-o localmente) como referência rápida no início de cada projeto novo.
  2. Percorra as categorias 1 e 2 antes de dar o primeiro prompt real ao agente.
  3. Percorra as categorias 3 a 6 antes de qualquer merge para a branch principal.
  4. Revise o checklist periodicamente — à medida que o projeto cresce, itens que pareciam opcionais (ex.: agente de segurança dedicado) tendem a se tornar obrigatórios.

Dicas e boas práticas

Dica: Adapte este checklist ao seu contexto — remova itens irrelevantes para projetos pequenos, mas nunca remova os itens de segurança (categoria 3) só para economizar tempo.
Dica: Transforme os itens mais críticos deste checklist em verificações automatizadas (lint de segredo, hook de pré-commit, gate de CI) sempre que possível — checklist manual é bom, checklist automatizado é melhor.
Dica: Revise este checklist com o time periodicamente — ele deve evoluir junto com as lições aprendidas do projeto, assim como o CLAUDE.md.

Erros comuns

Erro comum: Usar o checklist uma única vez no início do projeto e nunca mais revisitá-lo conforme a complexidade e o risco crescem.
Erro comum: Tratar os itens de segurança como opcionais em "projetos pequenos" — o tamanho do projeto não reduz o dano potencial de uma falha de autenticação ou vazamento de dados.

Exercício prático

Mão na massa: Aplique este checklist completo no próximo PR real gerado por um agente de IA no seu trabalho ou em um projeto pessoal. Marque item por item, honestamente, e identifique quais categorias você mais costuma pular — esse é o ponto exato onde vale investir mais disciplina daqui para frente.

Resumo do capítulo

Este checklist final consolida seis categorias — Planejamento, Configuração de agentes, Segurança, Qualidade, Contexto/Tokens e Deploy/DevOps — como referência rápida para qualquer projeto real com agentes de IA. Use-o como gate antes de iniciar uma tarefa e antes de fazer merge. Ele não substitui o entendimento profundo construído ao longo dos 14 capítulos deste curso, mas serve como lembrete prático e recorrente das disciplinas que separam um uso amador de um uso profissional e sustentável de agentes de IA no desenvolvimento de software.
