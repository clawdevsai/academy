9. Revisão de código, QA, segurança, DevOps e arquitetura com agentes especializados

Montando um pipeline de qualidade completo, com agentes, hooks e MCP trabalhando juntos.

🎯 Objetivos de aprendizagem

  • Mapear os papéis clássicos de um pipeline de qualidade: arquitetura, implementação, revisão, QA, segurança, DevOps.
  • Usar hooks como guardrails automáticos dentro desse pipeline (não apenas como conveniência).
  • Conectar agentes a ferramentas reais via MCP: GitHub, Jira/Linear, bancos de dados, ambientes de CI.
  • Montar um fluxo de PR (pull request) revisado por múltiplos agentes especializados em paralelo.
  • Saber onde a responsabilidade humana continua sendo insubstituível nesse pipeline.

Explicação conceitual

Este capítulo fecha o bloco intermediário juntando tudo: workflow estruturado (capítulo 5), múltiplos agentes especializados (capítulo 6), a mecânica de criar cada um (capítulo 7) e o rigor de especificar antes de implementar (capítulo 8). Agora aplicamos isso a um cenário muito comum em times profissionais: um pipeline de qualidade em torno de um PR (pull request) da user-api — nosso projeto FastAPI/SQLAlchemy de exemplo —, com papéis específicos e reforço automático de regras via hooks.

Um pipeline maduro normalmente separa sete papéis. Na user-api, cada um deles já é um subagente real em .claude/agents/, com tools e prompt restritos ao seu escopo, exatamente como você aprendeu no capítulo 7:

PapelAgenteResponsabilidadeTools/MCP típicos
Arquiteturaarquiteto-backAvalia impacto estrutural da mudança na arquitetura hexagonal, aponta acoplamentos entre domínio e adaptersRead, Grep, Glob
Implementaçãodev-backImplementa a partir de um plano aprovado (ex: endpoint POST /usuarios)Read, Write, Edit, Bash
Revisão de códigorevisorAponta bugs, más práticas, código duplicadoRead, Grep, Glob (sem Write/Edit)
QA / testesqaEscreve e roda testes pytest, valida cenários de borda (e-mail inválido, duplicado)Read, Write, Bash
Bug-fixbug-fixDiagnostica causa raiz a partir de logs/testes falhando e aplica a menor correção possívelRead, Write, Edit, Grep, Glob, Bash
Segurançacyber-secAudita vulnerabilidades, segredos expostos, dependências inseguras, falta de checagem de ownershipRead, Grep, Bash (scanners)
DevOps / infradevopsCI/CD (GitHub Actions), build da imagem Docker, observabilidadeBash, MCP de CI/infra

O papel de bug-fix é o elo que fecha o ciclo: quando revisão, QA ou segurança encontram algo que precisa de correção, é ele quem recebe a evidência (log, stack trace, teste falhando) e aplica a menor mudança possível para eliminar a causa raiz — nunca o próprio agente que encontrou o problema, para preservar a separação de papéis descrita abaixo.

A separação importa porque cada papel tem um viés natural diferente. Quem implementa tende a validar o próprio raciocínio; por isso o agente de revisão e o de QA devem ser entidades separadas, com contexto isolado, para atuarem como um segundo olhar de fato — o mesmo princípio de "grader separado" visto no capítulo 6.

Hooks entram aqui como guardrails automáticos, não apenas como atalhos de produtividade. Um hook PreToolUse pode bloquear um comando Bash perigoso (como rm -rf ou git push --force na main) antes mesmo que qualquer agente — implementador, DevOps ou outro — consiga executá-lo. Da mesma forma, um hook pode rodar uma checagem de segredos expostos (chaves de API, tokens) antes de qualquer commit ou push, funcionando como uma rede de segurança que independe da disciplina do agente naquele momento.

O MCP conecta esses agentes a sistemas reais de DevOps e QA: na user-api, o agente revisor pode comentar diretamente no Pull Request do GitHub que altera src/adapters/http/routes/usuarios.py, o devops pode consultar o status do workflow no GitHub Actions, e o cyber-sec pode abrir uma issue no Jira/Linear apontando a falta de checagem de ownership encontrada em rotas de tasks. Isso tira o pipeline do "só dentro do editor" e o coloca dentro do ciclo real de entrega do time.

Exemplos práticos

Hook simples de guardrail em .claude/settings.json da user-api, bloqueando comandos destrutivos antes da execução (o mesmo hook que orquestra a skill /pre-push-review):

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/scripts/bloquear-push-direto.sh"
          }
        ]
      }
    ]
  }
}

O script pode simplesmente recusar padrões conhecidos de risco:

#!/bin/bash
# .claude/scripts/bloquear-push-direto.sh
if echo "$CLAUDE_TOOL_INPUT" | grep -qE "rm -rf|push --force|push origin main|DROP TABLE"; then
  echo "Comando bloqueado por política de segurança da user-api." >&2
  exit 2   # crítico: só o código 2 bloqueia de fato a ação. exit 1 apenas registra um aviso e deixa passar.
fi

Agente de segurança dedicado do projeto user-api, .claude/agents/cyber-sec.md:

---
name: cyber-sec
description: >
  Use este agente antes de qualquer merge na user-api para auditar segredos
  expostos, dependências vulneráveis e padrões inseguros (ex: SQL sem
  parametrização via SQLAlchemy, falta de checagem de ownership em rotas).
tools: Read, Grep, Bash
---

Você é o engenheiro de segurança aplicativa (AppSec) da user-api.

Ao revisar um diff ou repositório:
1. Procure por segredos hardcoded (chaves, tokens, senhas, DATABASE_URL).
2. Rode e interprete `uv run pip-audit` (ou scanner equivalente) sobre as
   dependências do `pyproject.toml`.
3. Aponte padrões inseguros: queries SQLAlchemy montadas por concatenação de
   string, falta de validação Pydantic em `src/adapters/http/routes/`, e
   endpoints que não checam se o recurso pertence ao usuário autenticado
   (ex: `GET /tasks/{id}` sem checagem de ownership).
4. Classifique cada achado por severidade (crítico, alto, médio, baixo).
5. Nunca corrija o código diretamente — apenas reporte, com sugestão de fix.
   A correção fica a cargo do agente bug-fix, quando autorizado.

Agente de DevOps conectado a um MCP de CI, delegando deploy só após aprovação, .claude/agents/devops.md:

---
name: devops
description: >
  Use este agente para preparar pipelines de CI/CD da user-api (GitHub
  Actions), configurar build da imagem Docker e investigar falhas de
  build. Só deve disparar deploy em produção mediante confirmação
  explícita do humano.
tools: Bash, Read
---

Você cuida da esteira de CI/CD da user-api: uv sync, ruff, mypy, pytest e
build da imagem Docker.

- Ao investigar falha de build, leia os logs do GitHub Actions e aponte a
  causa raiz antes de sugerir qualquer mudança.
- Nunca dispare deploy em produção sem confirmação explícita na conversa.
- Prefira sempre rodar em ambiente de staging primeiro.

Fluxo de trabalho recomendado

  1. Configure hooks de guardrail antes de qualquer coisa — eles protegem mesmo quando um agente comete erro de julgamento.
  2. Ao abrir um PR (via MCP do GitHub ou manualmente), dispare em paralelo os agentes de revisão, QA e segurança.
  3. Consolide os achados dos três agentes em um único resumo — evite que o humano precise ler três relatórios separados.
  4. Só acione o agente de DevOps para preparar o deploy depois que revisão, QA e segurança aprovarem.
  5. Mantenha a decisão final de merge e de deploy em produção sempre com um humano.

Dica: trate hooks como a última linha de defesa, não a primeira — eles não substituem bons prompts e tools restritas, apenas garantem que erros não escapem mesmo quando tudo o mais falhar.
Dica: ao conectar agentes a MCP de GitHub/Jira, restrinja permissões de escrita (comentar é diferente de fechar issues ou aprovar PRs) até confiar plenamente no comportamento do agente.
Dica: use as skills review e security-review do Claude Code como ponto de partida pronto para revisão de PR e auditoria de segurança, antes de escrever agentes totalmente customizados do zero.

Erros comuns

Erro comum: deixar o agente de DevOps disparar deploy em produção automaticamente assim que os outros agentes aprovarem. Deploy em produção deve sempre ter um gate humano explícito, independentemente de quantos agentes já validaram o código.
Erro comum: confiar só nos hooks e achar que dispensa os agentes de revisão e segurança. Hooks pegam padrões conhecidos e simples (comandos perigosos, segredos óbvios); não substituem uma análise de lógica de negócio ou de vulnerabilidades mais sutis.
Erro comum: dar ao mesmo agente permissão de implementar e de aprovar a própria revisão de segurança/QA. Isso reintroduz o viés que a separação de papéis existe para evitar.
Mão na massa: na user-api (ou um repositório de teste equivalente), configure um hook PreToolUse que bloqueie git push --force. Depois crie os agentes revisor, qa e cyber-sec (você já viu exemplos nos capítulos 7 e 9). Abra um PR de teste no endpoint POST /usuarios com um problema proposital (ex: uma DATABASE_URL hardcoded em src/adapters/db/) e peça para os três agentes revisarem em paralelo. Confira se o cyber-sec pega o problema e se o hook bloquearia o push mesmo que ele passasse despercebido.

Resumo do capítulo

Um pipeline de qualidade profissional separa arquitetura, implementação, revisão, QA, segurança e DevOps em agentes especializados (na user-api: arquiteto-back, dev-back, revisor, qa, cyber-sec e devops, com bug-fix fechando o ciclo de correção), cada um com tools restritas ao seu papel. Hooks funcionam como guardrails automáticos — a última linha de defesa contra comandos perigosos ou segredos expostos, independente do julgamento do agente naquele momento. MCP conecta esses agentes a sistemas reais (GitHub, Jira/Linear, CI, bancos de dados), permitindo que o pipeline opere dentro do ciclo real de entrega do time. Mesmo com todo esse automatismo, decisões de merge e deploy em produção continuam exigindo um gate humano explícito. Isso encerra o bloco intermediário do curso — o próximo capítulo aprofunda como o contexto e os tokens funcionam por trás de tudo isso.
