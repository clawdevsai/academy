9. Revisão de código, QA, segurança, DevOps e arquitetura com agentes especializados

Montando um pipeline de qualidade completo, com agentes, hooks e MCP trabalhando juntos.

🎯 Objetivos de aprendizagem

  • Mapear os papéis clássicos de um pipeline de qualidade: arquitetura, implementação, revisão, QA, segurança, DevOps.
  • Usar hooks como guardrails automáticos dentro desse pipeline (não apenas como conveniência).
  • Conectar agentes a ferramentas reais via MCP: GitHub, Jira/Linear, bancos de dados, ambientes de CI.
  • Montar um fluxo de PR (pull request) revisado por múltiplos agentes especializados em paralelo.
  • Saber onde a responsabilidade humana continua sendo insubstituível nesse pipeline.

Explicação conceitual

Este capítulo fecha o bloco intermediário juntando tudo: workflow estruturado (capítulo 5), múltiplos agentes especializados (capítulo 6), a mecânica de criar cada um (capítulo 7) e o rigor de especificar antes de implementar (capítulo 8). Agora aplicamos isso a um cenário muito comum em times profissionais: um pipeline de qualidade em torno de um PR (pull request), com papéis específicos e reforço automático de regras via hooks.

Um pipeline maduro normalmente separa seis papéis. Cada um deles pode ser um subagente com tools e prompt restritos ao seu escopo, exatamente como você aprendeu no capítulo 7:

PapelResponsabilidadeTools/MCP típicos
ArquiteturaAvalia impacto estrutural da mudança, aponta acoplamentos e riscos de designRead, Grep, Glob
ImplementaçãoEscreve o código a partir de um plano aprovadoRead, Write, Edit, Bash
Revisão de códigoAponta bugs, más práticas, código duplicadoRead, Grep, Glob (sem Write)
QA / testesEscreve e roda testes, valida cenários de bordaRead, Write, Bash
SegurançaAudita vulnerabilidades, segredos expostos, dependências insegurasRead, Grep, Bash (scanners)
DevOps / infraCI/CD, configuração de deploy, observabilidadeBash, MCP de CI/infra

A separação importa porque cada papel tem um viés natural diferente. Quem implementa tende a validar o próprio raciocínio; por isso o agente de revisão e o de QA devem ser entidades separadas, com contexto isolado, para atuarem como um segundo olhar de fato — o mesmo princípio de "grader separado" visto no capítulo 6.

Hooks entram aqui como guardrails automáticos, não apenas como atalhos de produtividade. Um hook PreToolUse pode bloquear um comando Bash perigoso (como rm -rf ou git push --force na main) antes mesmo que qualquer agente — implementador, DevOps ou outro — consiga executá-lo. Da mesma forma, um hook pode rodar uma checagem de segredos expostos (chaves de API, tokens) antes de qualquer commit ou push, funcionando como uma rede de segurança que independe da disciplina do agente naquele momento.

O MCP conecta esses agentes a sistemas reais de DevOps e QA: um agente pode abrir e comentar em Pull Requests no GitHub, criar ou atualizar issues no Jira ou Linear, consultar um banco de dados de staging para validar uma migração, ou disparar um pipeline de CI e ler o resultado. Isso tira o pipeline do "só dentro do editor" e o coloca dentro do ciclo real de entrega do time.

Exemplos práticos

Hook simples de guardrail em .claude/settings.json, bloqueando comandos destrutivos antes da execução:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "scripts/bloquear-comandos-perigosos.sh"
          }
        ]
      }
    ]
  }
}

O script pode simplesmente recusar padrões conhecidos de risco:

#!/bin/bash
# scripts/bloquear-comandos-perigosos.sh
if echo "$CLAUDE_TOOL_INPUT" | grep -qE "rm -rf|push --force|DROP TABLE"; then
  echo "Comando bloqueado por política de segurança." >&2
  exit 2   # crítico: só o código 2 bloqueia de fato a ação. exit 1 apenas registra um aviso e deixa passar.
fi

Agente de segurança dedicado, .claude/agents/seguranca.md:

---
name: seguranca
description: >
  Use este agente antes de qualquer merge para auditar segredos expostos,
  dependências vulneráveis e padrões inseguros (ex: SQL sem parametrização,
  eval de input do usuário).
tools: Read, Grep, Bash
---

Você é um engenheiro de segurança aplicativa (AppSec).

Ao revisar um diff ou repositório:
1. Procure por segredos hardcoded (chaves, tokens, senhas).
2. Rode e interprete o scanner de dependências (`npm audit` ou equivalente).
3. Aponte padrões inseguros: SQL sem parametrização, deserialização insegura,
   uso de `eval`/`exec` com entrada de usuário.
4. Classifique cada achado por severidade (crítico, alto, médio, baixo).
5. Nunca corrija o código diretamente — apenas reporte, com sugestão de fix.

Agente de DevOps conectado a um MCP de CI, delegando deploy só após aprovação:

---
name: devops
description: >
  Use este agente para preparar pipelines de CI/CD, configurar deploy e
  investigar falhas de build. Só deve disparar deploy em produção mediante
  confirmação explícita do humano.
tools: Bash, Read
---

Você cuida da esteira de CI/CD do projeto.

- Ao investigar falha de build, leia os logs e aponte a causa raiz antes de
  sugerir qualquer mudança.
- Nunca dispare deploy em produção sem confirmação explícita na conversa.
- Prefira sempre rodar em ambiente de staging primeiro.

Fluxo de trabalho recomendado

  1. Configure hooks de guardrail antes de qualquer coisa — eles protegem mesmo quando um agente comete erro de julgamento.
  2. Ao abrir um PR (via MCP do GitHub ou manualmente), dispare em paralelo os agentes de revisão, QA e segurança.
  3. Consolide os achados dos três agentes em um único resumo — evite que o humano precise ler três relatórios separados.
  4. Só acione o agente de DevOps para preparar o deploy depois que revisão, QA e segurança aprovarem.
  5. Mantenha a decisão final de merge e de deploy em produção sempre com um humano.

Dica: trate hooks como a última linha de defesa, não a primeira — eles não substituem bons prompts e tools restritas, apenas garantem que erros não escapem mesmo quando tudo o mais falhar.
Dica: ao conectar agentes a MCP de GitHub/Jira, restrinja permissões de escrita (comentar é diferente de fechar issues ou aprovar PRs) até confiar plenamente no comportamento do agente.
Dica: use as skills review e security-review do Claude Code como ponto de partida pronto para revisão de PR e auditoria de segurança, antes de escrever agentes totalmente customizados do zero.

Erros comuns

Erro comum: deixar o agente de DevOps disparar deploy em produção automaticamente assim que os outros agentes aprovarem. Deploy em produção deve sempre ter um gate humano explícito, independentemente de quantos agentes já validaram o código.
Erro comum: confiar só nos hooks e achar que dispensa os agentes de revisão e segurança. Hooks pegam padrões conhecidos e simples (comandos perigosos, segredos óbvios); não substituem uma análise de lógica de negócio ou de vulnerabilidades mais sutis.
Erro comum: dar ao mesmo agente permissão de implementar e de aprovar a própria revisão de segurança/QA. Isso reintroduz o viés que a separação de papéis existe para evitar.
Mão na massa: em um repositório real (ou de teste), configure um hook PreToolUse que bloqueie git push --force. Depois crie os agentes revisor-codigo, qa-tester e seguranca (você já viu exemplos nos capítulos 7 e 9). Abra um PR de teste com um problema proposital (ex: uma chave de API hardcoded) e peça para os três agentes revisarem em paralelo. Confira se o agente de segurança pega o problema e se o hook bloquearia o push mesmo que ele passasse despercebido.

Resumo do capítulo

Um pipeline de qualidade profissional separa arquitetura, implementação, revisão, QA, segurança e DevOps em agentes especializados, cada um com tools restritas ao seu papel. Hooks funcionam como guardrails automáticos — a última linha de defesa contra comandos perigosos ou segredos expostos, independente do julgamento do agente naquele momento. MCP conecta esses agentes a sistemas reais (GitHub, Jira/Linear, CI, bancos de dados), permitindo que o pipeline opere dentro do ciclo real de entrega do time. Mesmo com todo esse automatismo, decisões de merge e deploy em produção continuam exigindo um gate humano explícito. Isso encerra o bloco intermediário do curso — o próximo capítulo aprofunda como o contexto e os tokens funcionam por trás de tudo isso.
