4. Prompts, Comandos, Agentes, Tools, MCPs, Skills e Hooks — as diferenças

Organizando o vocabulário: qual peça faz o quê no ecossistema do Claude Code, e como elas se encaixam.

🎯 Objetivos de aprendizagem

  • Distinguir com clareza sete termos que costumam ser confundidos: prompts, comandos, agentes, tools, MCPs, skills e hooks.
  • Entender a relação de granularidade entre esses conceitos — do mais específico (tool) ao mais orquestrador (agente).
  • Saber quando usar um comando de barra (slash command) em vez de escrever um prompt do zero.
  • Compreender como hooks funcionam como guardrails automáticos de segurança.

Explicação conceitual

Até aqui você já viu prompts, agentes e subagentes. Este capítulo fecha o vocabulário essencial, porque nos próximos capítulos do curso esses termos vão aparecer sem explicação repetida. A confusão mais comum de quem está começando é achar que tudo isso é "a mesma coisa com nomes diferentes" — não é. Cada termo descreve uma camada diferente de como o Claude Code funciona.

Pense numa analogia com uma cozinha profissional:

  • O prompt é o pedido do cliente ("quero um risoto de limão siciliano").
  • As tools são os utensílios individuais que o cozinheiro usa (faca, fogão, liquidificador) — a ação mais básica e granular possível.
  • Uma skill é uma receita documentada que o cozinheiro segue para não errar um prato específico.
  • Um comando (slash command) é um atalho no cardápio — pedir "o de sempre" em vez de descrever o prato inteiro toda vez.
  • Um agente é o próprio cozinheiro (ou um subagente, um ajudante especializado): decide quais utensílios usar, em que ordem, seguindo ou não uma receita.
  • Um MCP é a ponte com fornecedores externos — o telefone que liga para o mercado, o distribuidor de peixe, o serviço de entrega.
  • Um hook é a regra de segurança da cozinha que roda sozinha, sem depender do cozinheiro lembrar — por exemplo, "sempre lave as mãos antes de tocar em carne crua", verificado automaticamente.

Agora, com definições técnicas mais precisas:

PromptInstrução em linguagem natural dada pelo usuário ao agente.Entrada do usuário
ToolUma capacidade individual e granular que o agente pode invocar (ex.: Read, Write, Bash, WebSearch).Mais granular

Um detalhe importante: em 2026, comandos de barra customizados (slash commands) foram unificados com Skills. Ou seja, quando você cria uma skill, ela automaticamente vira invocável via /nome-da-skill — não são mais dois sistemas separados como eram no passado.

Outro ponto que costuma confundir: quando você conecta um servidor MCP (por exemplo, um MCP do GitHub), os prompts que esse servidor expõe também aparecem como comandos de barra, no formato /mcp__nome-do-servidor__nome-do-prompt. Ou seja, MCP e comandos se cruzam, mas a origem é diferente: um vem de um serviço externo conectado via protocolo, o outro é definido localmente no seu projeto.

Exemplos práticos

Tool em ação (nível mais granular, geralmente invisível para você — o agente decide sozinho usá-la):

[Claude decide internamente]
Tool: Read
Input: { "file_path": "src/api/pagamentos.ts" }

Skill / comando — um arquivo SKILL.md que vira o comando /gerar-changelog:

---
name: gerar-changelog
description: Gera um changelog em português a partir dos commits desde a última tag.
---

1. Rode `git log` desde a última tag até HEAD.
2. Agrupe os commits por tipo (feat, fix, chore).
3. Escreva um changelog em português, em Markdown,
   seguindo o padrão Keep a Changelog.

Uso no terminal:

> /gerar-changelog

MCP — conectando um servidor externo (ex.: GitHub) e usando um dos prompts que ele expõe:

# depois de conectar o servidor MCP do GitHub
> /mcp__github__criar_pull_request

Hook — um guardrail que bloqueia comandos perigosos de shell, configurado em .claude/settings.json, rodando antes de qualquer uso da tool Bash:

#!/bin/bash
# hook PreToolUse — lê o input da tool via stdin
INPUT=$(cat)
COMANDO=$(echo "$INPUT" | jq -r '.tool_input.command')

if echo "$COMANDO" | grep -qE 'rm -rf|DROP TABLE'; then
  echo "Comando bloqueado por política de segurança." >&2
  exit 2   # código 2 bloqueia a ação
  # Atenção: exit 1 NÃO bloqueia — apenas registra um aviso e a ação segue normalmente.
  # Só exit 2 impede a execução da ferramenta. É um erro comum usar exit 1 achando que é seguro.
fi
exit 0

Esse hook roda automaticamente, sem depender do agente "lembrar" de ser cuidadoso — é uma camada de segurança determinística, fora do controle do modelo de linguagem.

Fluxo de trabalho recomendado

  1. Comece só com prompts diretos — é o jeito mais simples de aprender a "conversar" com o agente (Capítulos 1 e 2).
  2. Quando perceber que repete o mesmo tipo de pedido, transforme-o em uma skill/comando de barra.
  3. Quando uma tarefa exigir "modo de pensar" isolado, crie um subagente especializado (Capítulo 3).
  4. Quando precisar de dados ou ações fora do seu repositório (ex.: abrir uma issue no GitHub), conecte um MCP.
  5. Quando precisar de uma regra de segurança que nunca pode falhar (ex.: nunca commitar segredos), use um hook — não confie só em instrução de prompt para isso.

Dicas e boas práticas

Dica: não tente decorar as diferenças de uma vez — use a analogia da cozinha (prompt = pedido, tool = utensílio, skill = receita, agente = cozinheiro, MCP = fornecedor externo, hook = regra de segurança automática) sempre que precisar se situar.
Dica: prefira resolver com skills/comandos antes de criar um subagente novo — subagentes têm mais overhead de configuração e só compensam quando há real necessidade de isolar contexto ou restringir ferramentas.
Dica: regras de segurança inegociáveis (nunca apagar produção, nunca commitar chave de API) devem virar hooks, não apenas instruções em texto — hooks são determinísticos, prompts são probabilísticos.

Erros comuns

Erro comum: confundir skill com agente. Uma skill é um procedimento que um agente segue; ela não decide nada sozinha, não tem autonomia própria — quem decide usá-la (ou não) é o agente.
Erro comum: achar que MCP é só "mais uma tool". Um MCP pode expor várias tools, prompts e recursos de uma vez — é uma camada de integração, não uma ferramenta única.
Erro comum: depender só de instruções em linguagem natural ("por favor nunca rode comandos destrutivos") para segurança crítica. Modelos de linguagem podem falhar em seguir instruções em casos extremos; hooks são a camada que garante isso de forma mecânica.

Exercício prático

Mão na massa: escolha uma tarefa que você repetiria com frequência no seu projeto (ex.: gerar um resumo de pull request, ou rodar um checklist antes de commitar). Escreva-a como uma skill simples (arquivo SKILL.md com name, description e os passos), salve na pasta apropriada do projeto, e teste invocando com /nome-da-skill no Claude Code. Depois, identifique uma regra de segurança do seu projeto que você gostaria de garantir sempre — e escreva, ao menos em pseudocódigo, como seria um hook para isso.

Resumo do capítulo

Prompts, tools, skills, comandos, agentes, MCPs e hooks são camadas diferentes e complementares do ecossistema do Claude Code: tools são a ação mais granular; skills documentam procedimentos e, desde a unificação em 2026, viram automaticamente comandos de barra; agentes orquestram tudo isso para cumprir um objetivo; MCPs conectam o agente a serviços externos; e hooks garantem regras de segurança de forma automática e determinística, independente do que o modelo "decide" fazer. Entender essa hierarquia evita confusão e ajuda a escolher a ferramenta certa para cada problema nos próximos capítulos do curso.
