11. Boas práticas para projetos reais

Do laboratório para produção: como integrar agentes de IA em um fluxo de engenharia de software sério.

🎯 Objetivos de aprendizagem

  • Manter um CLAUDE.md vivo, atualizado e útil ao longo do tempo.
  • Versionar prompts, configs de agentes e specs junto com o código-fonte.
  • Revisar diffs gerados por IA com o mesmo rigor de um Pull Request humano.
  • Usar testes automatizados como rede de segurança para mudanças feitas por agentes.
  • Escalar de "um agente, uma tarefa" para orquestração multi-agente de forma gradual e segura.

Explicação conceitual

Até aqui você aprendeu as peças técnicas: subagentes, agentes customizados, Spec-Driven Development, agentes de QA/segurança/DevOps. A pergunta deste capítulo é diferente: como isso tudo se comporta dentro de um projeto real, com histórico, com outros desenvolvedores, com prazos e com produção rodando? A resposta não é sobre técnica de prompt — é sobre disciplina de engenharia aplicada ao uso de IA.

A analogia mais útil é tratar o agente como um desenvolvedor júnior muito rápido, mas sem julgamento de longo prazo. Ele escreve código plausível em segundos, mas não tem instinto sobre as decisões arquiteturais tomadas seis meses atrás, não sabe por que aquele hack "temporário" ainda está lá, e não vai necessariamente notar que uma mudança pequena quebra um contrato implícito em outro módulo. Boas práticas de projeto real existem justamente para compensar essa lacuna de julgamento — com processo, e não com confiança cega.

Exemplos práticos

Um CLAUDE.md de projeto real não é um "prompt engenhoso" — é um documento de engenharia, versionado no repositório, revisado como qualquer outro arquivo: 

# CLAUDE.md

## Stack
- Backend: Node.js 20 + Express + Prisma + PostgreSQL
- Frontend: React + Vite + TailwindCSS
- Testes: Vitest (unitário), Playwright (E2E)

## Convenções
- Toda rota de API precisa de validação com Zod antes do handler.
- Erros de negócio usam a classe AppError (não lançar Error genérico).
- Nunca commitar migrations sem rodar `npm run test:db` antes.

## Comandos úteis
- `npm run dev` — sobe o ambiente local
- `npm run test` — roda toda a suíte
- `npm run lint:fix` — corrige lint automaticamente

## Decisões arquiteturais
- Optamos por autenticação via JWT + refresh token em cookie httpOnly
  (ADR-003, ver docs/adr/003-auth.md) em vez de sessão no servidor,
  por causa do requisito de múltiplos clientes (web + mobile).

Note o último bloco: "Decisões arquiteturais". Isso é o que evita que um agente (ou um novo desenvolvedor humano) proponha "vamos trocar para sessão no servidor" sem saber que essa opção já foi avaliada e descartada por um motivo concreto.

Sobre versionar prompts e configs: se você tem agentes customizados (capítulo 6) com instruções específicas do domínio, esses arquivos de definição de agente pertencem ao repositório, não à sua máquina local: 

projeto/
  .claude/
    agents/
      revisor-seguranca.md
      qa-automatizado.md
    CLAUDE.md
  src/
  tests/

Assim, qualquer pessoa do time — ou o próprio agente em uma sessão futura — usa exatamente a mesma configuração, e mudanças nesses arquivos passam por revisão de código como qualquer outra mudança de comportamento do sistema.

Fluxo de trabalho recomendado

  1. Antes de pedir a primeira tarefa a um agente em um projeto novo, escreva (ou peça ao agente para gerar e você revisar) um CLAUDE.md inicial.
  2. Comece com uma tarefa isolada e um único agente. Só introduza orquestração multi-agente quando o fluxo simples já estiver confiável.
  3. Toda mudança gerada por agente passa por revisão de diff — leia como leria um PR de um colega novo no time.
  4. Rode a suíte de testes antes de aceitar qualquer mudança; se não existe suíte, esse é o primeiro investimento a fazer.
  5. Quando uma decisão importante for tomada (arquitetura, biblioteca, padrão), registre-a em um ADR curto e referencie no CLAUDE.md.

Dicas e boas práticas

Dica: Trate atualizações do CLAUDE.md como parte do "definition of done" de qualquer tarefa que introduza uma convenção nova ou decisão arquitetural.
Dica: Peça ao próprio agente para sugerir atualizações ao CLAUDE.md depois de uma tarefa complexa — muitas vezes ele identifica lacunas de contexto que você não percebeu.
Dica: Escale a orquestração multi-agente igual escala microsserviços: só quando a complexidade real justificar, não porque parece mais sofisticado.

Erros comuns

Erro comum: Ter um CLAUDE.md que foi escrito uma vez no início do projeto e nunca mais atualizado — vira documentação mentirosa, pior do que não ter nada.
Erro comum: Revisar diffs de IA "por cima", com menos rigor do que se revisaria um PR humano, sob a premissa errada de que "a IA não erra tanto assim".
Erro comum: Pular direto para orquestração multi-agente complexa em um projeto que ainda nem tem testes automatizados — a complexidade de coordenação sem rede de segurança é uma receita para bugs difíceis de rastrear.

Exercício prático

Mão na massa: Pegue um projeto real (ou o projeto do capítulo 13 mais adiante) e escreva um CLAUDE.md do zero com pelo menos quatro seções: Stack, Convenções, Comandos úteis e Decisões arquiteturais. Depois, peça a um agente para executar uma tarefa pequena usando apenas esse arquivo como contexto — sem você repetir nada no prompt — e avalie se o resultado seguiu as convenções descritas.

Resumo do capítulo

Projetos reais exigem disciplina de engenharia, não apenas técnica de prompt. Um CLAUDE.md vivo e atualizado, prompts e configs versionados junto com o código, revisão rigorosa de diffs gerados por IA, testes automatizados como rede de segurança e uma escalada gradual de complexidade (de um agente para múltiplos) são os pilares que tornam o uso de agentes sustentável em produção, e não apenas impressionante em uma demo.
